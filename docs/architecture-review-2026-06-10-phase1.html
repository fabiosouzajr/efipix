Hoist the idempotency protocol behind one replay seam

internal/charge/api/handler.go  ·  internal/platform/idempotency/idempotency.go

Before — shallow

Charge intake handler · create()

  • read Idempotency-Key → 400 if absent
  • buffer body · sha256 fingerprint
  • Reserve → switch replay / conflict / inflight
  • parse · validate · resolve PixKey · use case
  • SaveResult(status, body)

idempotency.Store

Reserve() · SaveResult()

Amber = the replay protocol, smeared through the handler. The Store seam holds two methods; the rules that give them meaning live outside it.

After — deep

Idempotent replay · deep module

key + fingerprint → at-most-once.
replay / conflict / inflight + status codes + body capture (now internal)

Do(ctx, key, fp, fn) (status, body)

Charge intake handler

parse · validate · use case · respond

idempotency.Store (unchanged adapter)

The protocol becomes a deep module the handler wraps. Handler shrinks to four verbs.

Problem

The replay protocol (fingerprint, Reserve, the four-state switch, status codes, result capture) is interleaved inside the Charge intake handler; the Store seam exposes only Reserve/SaveResult.

Solution

Pull the protocol behind one interface — a replay decorator or Gin middleware — so any write handler runs at-most-once per key without re-copying the dance.

Wins

  • locality: replay rules in one module
  • leverage: Refund · cancel · CobV reuse it
  • interface shrinks: handler loses 30 lines
  • protocol becomes the test surface, not each e2e
The overview already names this an “idempotency middleware” (file 05 line 23) — the plan’s implementation quietly inlined it into the handler. This deepening just delivers the module the plan promised.

Resolve the acting account once — stop the PixKey leak

The Charge intake module reaches into the Tenant module mid-request for the PixKey. Resolution is two-thirds in the middleware, one-third in the handler — across three transactions.

After — one resolution, one join

flowchart TB
  MW["Account resolver middleware"] -->|"tx1 admin"| R1["api-key → tenant"]
  MW -->|"tx2 tenant — one join"| R2["provider ⨝ pix_keys"]
  MW --> CTX["Resolved\n{TenantID, ProviderID, PixKey}"]
  CTX --> H["Charge intake handler\n(reads res.PixKey)"]
  classDef deep fill:#0f172a,stroke:#0f172a,color:#e2e8f0;

The acting account (Tenant + PaymentProvider + its PixKey) resolves once, behind the resolver. The handler reads res.PixKey and never touches the Tenant module.

Problem

Account resolution is smeared: middleware resolves (Tenant, PaymentProvider); the Charge handler separately calls PixKeyFor. Four narrow repository lookups, three transactions per create.

Solution

Resolve the whole acting account once and carry PixKey in tenantctx.Resolved. One provider ⨝ pix_keys query replaces two lookups.

Wins

  • locality: account resolution in one module
  • leverage: 3 transactions → 2
  • Charge intake stops knowing about PixKeys
  • per-request seam crossing disappears
Touches a locked signature: Resolved{TenantID, ProviderID} gains PixKey — update the overview’s locked-signatures block first. Not an ADR-0003 conflict: that ADR specifies tenant+provider resolution only; the PixKey is additive.

Short interface, tall implementation = deep. Reading payment_events now explains the Charge lifecycle.

Problem

appendEvent always writes {}; MarkFailed(reason) takes a reason and never persists it. The audit interface advertises more than the implementation delivers — ADR-0002 and CONTEXT make PaymentEvent the audit history, but the “failed” row records nothing.

Solution

appendEvent takes a structured payload; transitions record their facts (failure reason, provider location/QR on activate). Event construction stays in the entity.

Wins

  • locality: event shaping in the entity
  • leverage: the audit trail explains itself
  • interface stops lying about what it stores
  • satisfies the ADR-0002 audit intent
Half depth-smell, half latent bug: the Phase-1 exit checklist asserts a “failed” audit row exists — it does, but it is contentless. Worth a buglog entry alongside the refactor.
Worth exploring in-process candidate 4

One error-response seam — and stop leaking provider text

internal/platform/httpx/errors.go · internal/charge/api/handler.go

Before — rendering scattered per branch

StatusFor(err) — status only
gin.H{"error": err.Error()} leaks internals
mustJSON(gin.H{"error": ...})
c.Data(status, ..., body)
c.JSON(StatusFor(err), gin.H{...})
"provider...: efi down" → client

Every handler branch hand-rolls the body. Wrapped internal text — the EFí PixProvider’s “efi down” — reaches the caller.

After — one consolidated band

httpx.WriteError(c, err)

  • · Kind → status
  • · safe client body (no internal cause)
  • · logs the wrapped cause server-side

handlers return err · stop shaping bodies

Status, client-safe shape, and logging concentrate in one deep module the handlers call once.

Problem

StatusFor maps Kind→status, but body shaping is hand-rolled in every branch, and err.Error() ships wrapped internal text to clients.

Solution

A single WriteError(c, err) (or Gin error middleware) owns status + client-safe body + server-side logging of the cause.

Wins

  • locality: one place renders errors
  • leverage: every handler reuses it
  • interface shrinks: handlers return err
  • provider/internal detail stays server-side

Top recommendation

Start with candidate 1 — the idempotency replay seam

Highest leverage and lowest risk: it concentrates a protocol that every future write endpoint (Refund, cancel, CobV) will otherwise re-copy, it aligns the build with the overview’s own stated “idempotency middleware”, and it shrinks the fattest module in the slice. The EFí adapter’s efiClient already proves this team builds deep, two-adapter seams (fake in tests, SDK in prod) — apply the same move one layer up, to idempotency.

Then 2 and 4 (both shrink the fat Charge intake handler); 3 is small and fixes a latent audit gap — fold it into file 04 before the slice ships.
